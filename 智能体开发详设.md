# TradeFlow-AI · 9 智能体开发详细拆解（WBS）

> 这份是**详细设计 / 任务分解**，入门与架构见 `README.md` / `ROADMAP.md`；这份是把每个智能体拆到"可开工"的粒度。开工顺序见《开发落地方案.md》。
> 每个智能体不是"一句话功能"，而是**一组子功能 + 一份专属数据清单 + 一批要开发的工具**。下面逐个拆。

---

## 阅读方式

每个智能体按统一模板写：

- **定位** —— 它在整个系统里干嘛、被谁用
- **核心功能拆解** —— 把老板原话里的功能拆成一条条**可交付的子任务**（每条能单独开发、单独验收）
- **要开发的工具 (Tool)** —— 需要写代码的部分（放 `tradeflow/tools/`）
- **人设 (Prompt) 要点** —— 岗位说明书里必须写清的东西
- **判断规则 (Skill)** —— 需要沉淀的 SOP / 阈值 / 分级逻辑（放 `skills/<名>/`）
- **需要的数据 (Data)** —— 老板要提供、丙要整理的原始资料（放 `data/<名>/`）
- **依赖** —— 依赖哪些别的智能体 / query-system / SP-API
- **输出物** —— 交付什么
- **难度 / 梯队 / 负责人 / 验收标准**

---

## 第 0 部分：底层基础设施（甲，第 1 周先做，是所有智能体的地基）

这些不是业务智能体，但**不做完，乙丙的智能体没法规范化、没法串成流水线**。属关键路径。

| 编号 | 任务 | 说明 | 验收 |
|---|---|---|---|
| 0.1 | 智能体模板 + 人设组合机制 | 规范"每个智能体 = 读 `prompts/<名>.md` + 拼 `skills/<名>/*.md` + 挂 tools"的固定套路；每个智能体配一个 `examples/run_<名>.py` 一键脚本 | 新建一个智能体只需加 3 个文件，不改引擎 |
| 0.2 | 智能体注册表 + 编排 | 让智能体能**调用另一个智能体**（如 Listing 生成后自动调 合规风控 过一遍；选品调 市场分析+拆解+合规） | 能写出"A 调 B"的流水线并跑通 |
| 0.3 | 数据加载层 | 统一读 `data/<名>/` 下的 CSV/Excel/md（禁词表、词根库、报表、ASIN 清单），给工具和 prompt 用 | 一个函数 `load_data(agent, name)` 能读表格返回结构化数据 |
| 0.4 | 切换真实模型并上线 | provider 从 mock → bailian/anthropic，端到端验证真实输出；对接部署 | 真实模型能出正常业务结果 |
| 0.5 | query-system 补接口 | 现在只有 `/api/search`（关键词→TOP N）。#5 拆解需要"按 ASIN 抓单个产品全貌"，要加 `/api/product/{asin}`。**复用现有 `models.py`（已有 `parent_id`/`variant_attributes` 变体字段）与评论情感分析，只加路由，勿重建数据模型** | 输入 ASIN 返回完整 Listing+变体+评价 |

## 第 0.6 部分：工程规范（全局强制，开工前置）

> 这些不是某个智能体的功能，而是**所有智能体共用的工程底线**。缺了它们，AI 输出质量无法保证、后期必然返工。前三条（评测集 / 数据 schema / 阈值）属**开工前置**：未满足不进入编码。

**G1 效果评测集（eval / golden set）—— 每个智能体必配**
- 输出是 LLM 生成的，验收不能"跑一次人看"就算过。每个智能体配 `tests/golden/<名>.jsonl`（输入 → 期望判定），做成可回归评测。
- #1 合规尤其要有**"必拦 N 条 + 必放行 N 条"**固定测试集，换模型 / 改 prompt / 更新数据后一键回归，防退化。

**G2 数据 schema 先行 —— 0.3 的隐含前置**
- 《数据总清单》只说"要什么"，不够。每类数据必须补一张**表结构定义（列名 / 类型 / 示例行）**，否则丙整理与甲解析必然对不上。
- `load_data(agent, name)` 按 schema 校验，缺列 / 格式错要显式报错而非静默。

**G3 判断规则量化前置**
- #4 分类阈值、#7 评分权重等"待定"项，必须**与老板对齐成明确数字后才开工**，写成对应智能体的准入闸门。

**G4 编排失败处理（0.2 必含）**
- query-system 单次可能几分钟（`_SEARCH_TIMEOUT=300`）。链路（如 #7 串 #1→#5→#6）中某环超时 / 报错时，须定义**降级 / 重试 / 部分结果**策略，不得整链崩溃。

**G5 数据与密钥安全**
- SP-API token 等密钥进 env / 密钥库，不入代码、不入日志。
- #8 买家消息含 PII、#4/#9 含店铺财务：敏感数据不落公共缓存，访问受控。

**G6 v1 范围锁定**
- 明确 v1 只做哪几个站点 / 哪几种语言（#1 分站点禁词、#2 多站点文案都受此约束），其余排期后置，防 scope creep。

**G7 被编排的智能体必须有结构化输出契约**
- #7 要靠 0.2 编排**程序化消费** #1/#5/#6 的结果，散文报告没法稳定解析。凡是**被别人调用**的智能体（#1/#5/#6，及被 #7#8 调的 #1），除给人看的报告外，必须同时返回**机器可读 JSON**（如 #6 输出 `{体量, 竞争强度, 价格带[], 季节性, 风险等级}`），并固定字段名，供上游直接取用。

**建议开发顺序修正**：0.1 模板与 #1 合规是**鸡生蛋**关系——0.1 搭骨架的同时用 #1 做第一个"垂直切片"互相校准，而非等 0.1~0.4 全部做完再动 #1。详见《开发落地方案.md》。

## 智能体依赖关系与开发顺序（先看全局）

```
              ┌─────────────────────────────────────────┐
              │   #1 合规风控（底层守门员，最先做）        │  ← 被 #2 #3 #5 #6 #7 #8 强制调用
              └─────────────────────────────────────────┘
第一梯队(无API)   #2 Listing文案      #3 图文视频提示词      （靠 #1 + 数据，丙主导）
第二梯队(Excel)   #4 广告优化                                （乙）
第三梯队(选品)    #5 爆款拆解 ─┐  #6 市场分析 ─┐             （乙，都靠 query-system）
                            └──────────────┴──▶ #7 智能选品  （甲，综合 #1#5#6+成本表）
第四梯队(SP-API)  #8 评价&客服      #9 库存/利润/定价          （甲，后期，需授权）
```

**开发顺序**：0.3 数据层 → 0.5 产品接口 →（0.1 模板 与 #1 合规**交叉进行**，用 #1 当第一个垂直切片校准模板，见 0.6 修正）→ 0.4 切真实模型 → 0.2 编排 → #2/#3（丙并行）→ #4 → #5/#6 → #7 → #8/#9。

## #1 合规风控智能体 【底层 / 全局 / 乙做甲 review / 难度：中】

**定位**：全局底层守门员，**所有涉及文案和选品的智能体都强制调用它**。优先开发。

**核心功能拆解**（每条 = 一个子任务）：
- 1.1 **极限词/医疗夸大词拦截** —— 扫描文案，命中禁词表则拦截并指出位置
- 1.2 **分站点禁词** —— 不同国家站点禁词不同（美/英/德/日…），按 `site` 参数选对应表
- 1.3 **侵权品牌/IP/专利描述过滤** —— 匹配"自有品牌+竞品 IP 黑名单"，识别他人商标、专利话术
- 1.4 **合规替代建议** —— 命中违规词时，给出合规的替代表达（不只是报错）
- 1.5 **选品阶段类目风险标注** —— 输入类目，返回"高审核/高侵权风险"等级标记（小红旗预警）
- 1.6 **对外服务化** —— 封装成统一入口 `compliance_gate(text/category, site)`，供别的智能体一行调用
- 1.7 **白名单 / 人工申诉出口** —— "宁可误伤"必须配白名单（自有品牌词、已确认合规的表达），否则误伤无出口。命中白名单直接放行并记录覆盖日志

**要开发的工具 (Tool)**：
- `check_forbidden_words(text, site)` —— 把现有 5 词的桩改成读 `data/compliance/禁词表_<站点>.csv`
- `match_ip_brand_patent(text)` —— 黑名单/商标/专利匹配
- `flag_category_risk(category)` —— 类目风险等级
- `compliance_gate(...)` —— 组合上面三个的统一入口

**人设 (Prompt) 要点**：你是合规风控专家，宁可误伤不可漏放；输出必须给"违规点 + 原因 + 合规替代"。

**判断规则 (Skill)**：风险分级标准（高/中/低）、把"历史侵权/下架复盘案例"转成可执行的排查规则清单。

**需要的数据 (Data)**：
- 各国站点禁词表（按站点分文件）
- 自有品牌 + 竞品 IP 黑名单
- 各类目审核禁止描述
- 历史侵权/违规下架复盘案例
- 白名单（自有品牌词 / 已确认合规表达）

**依赖**：无（最底层）。**被 #2 #3 #5 #6 #7 #8 调用。**
**输出物**：`{passed, 违规点[], 原因, 合规替代[], 类目风险等级}`
**验收标准**：给一段含违规词/侵权词的文案，能 100% 拦截、指出每个违规点并给合规替代；给一个高风险类目能正确标红。

## #2 Listing 文案生成智能体 【第一梯队 / 丙主导 / 难度：低-中】

**定位**：产品上新时自动产出全套多站点文案。第一梯队，无需任何 API。

**核心功能拆解**：
- 2.1 **标题生成**（多站点）
- 2.2 **五点描述生成**（5 条）
- 2.3 **后台 Search Terms 生成**
- 2.4 **A+ 页面短文案生成**
- 2.5 **广告短句生成**
- 2.6 **QA 生成**
- 2.7 **三套写作逻辑切换**：新品 / 成熟品 / 清仓（写法不同）
- 2.8 **自动规避违规**：生成后调用 #1 合规风控过一遍，命中就重写。**须设最大重试次数**（如 3 次），超限则升级人工，避免与 #1"宁可误伤"之间死循环
- 2.9 **关键词词根注入**：从词根库里挑词，合理埋进标题和五点

**要开发的工具 (Tool)**：
- 调用 #1 的 `compliance_gate`（复用，不重写）
- `inject_keywords(text, root_lib)` —— 从 `data/listing/关键词词根库.csv` 取词注入（甲/乙帮丙写这个工具）

**人设 (Prompt) 要点**：见 `prompts/listing.md`（**待 0.1 模板机制产出**，届时丙维护；当前尚不存在）。
**判断规则 (Skill)**：标题公式、五点结构、三套写作逻辑差异、爆款范例——见 `skills/listing/写作规则.md`。

**需要的数据 (Data)**：
- 全店铺产品参数库
- 爆款 Listing 范文（越多越好，AI 靠例子学）
- 完整关键词词根库
- 标准化上新 SOP
- 产品卖点 & 差评痛点清单

**依赖**：#1 合规风控、词根库。
**输出物**：一套完整文案（Title/Bullets/Search Terms/A+/广告短句/QA），0 违规词。**输出语言随站点**（如美国站出英文），语种范围受 G6 约束。
**验收标准**：输入产品参数 → 产出全套文案，合规 100% 通过，核心关键词覆盖到位，三种品况写法有明显区别。

## #3 图文视频提示词智能体 【第一梯队 / 丙 / 难度：低-中】

**定位**：为产品图、A+、短视频出"创意提示词"和脚本。第一梯队，无需 API。
**v1 边界**：只产出**提示词文本 + 脚本**（供人贴到 Midjourney / 即梦 / 剪辑），**不接图像 / 视频生成 API**；接生成后置为后续版本。

**核心功能拆解**：
- 3.1 **AI 绘图提示词生成**（中英双语，适配亚马逊主图/场景图规范）
- 3.2 **竞品图片构图配色拆解** —— 输入竞品图文样例，输出构图/配色分析 + 优化建议
- 3.3 **短视频脚本生成** —— 开箱 / 测评两种脚本
- 3.4 **图片平台规范校验** —— 对照官方规范检查（主图白底、无水印文字等）

**要开发的工具 (Tool)**：主要靠 prompt+规范文档，工具较少；可选 `check_image_rules(描述)` 对照规范清单。
**人设 (Prompt) 要点**：你是亚马逊视觉/短视频创意专家，产出可直接给设计/剪辑用的提示词和脚本。
**判断规则 (Skill)**：亚马逊图文视频官方规范转成 checklist；优质构图配色套路；短视频脚本结构模板。

**需要的数据 (Data)**：
- 产品参数与拍摄避雷点
- 优质竞品图文样例
- 亚马逊图文视频官方规范
- 成熟短视频脚本案例

**依赖**：规范文档（可选调 #1 校验文字）。
**输出物**：绘图提示词（中英）+ 构图优化建议 + 短视频脚本 + 规范检查结果。
**验收标准**：输入产品 → 产出可用的绘图 prompt 和短视频脚本，规范校验能抓出明显违规。

## #4 广告优化分析智能体 【第二梯队 / 乙 / 难度：中-高】

**定位**：导入广告 Excel 报表，替代人工筛词、控 ACOS。只需导表，不用 API。

**核心功能拆解**：
- 4.1 **报表导入解析** —— 支持 SP / SD / SB 三种报表（结构不同，各写解析）
- 4.2 **指标统计** —— 曝光、点击、CTR、CPC、转化率、花费、销售额、ACOS，按关键词/广告活动/ASIN 汇总
- 4.3 **关键词三分类** —— 高转化好词 / 潜力长尾词 / 无效垃圾词（按阈值规则）
- 4.4 **竞价 & 预算调整建议** —— 基于"人工调词控 ACOS 规则"给每个词加/减价、调预算
- 4.5 **批量否定词生成** —— 无效词导出成可直接上传的否定词清单
- 4.6 **拓词/埋词** —— 从好词的词根拓展新词，补充埋词库

**要开发的工具 (Tool)**：
- `parse_ad_report(excel_path, report_type)` —— 用 openpyxl 读 Excel（query-system 已装该库）
- `classify_keywords(rows, rules)` —— 按阈值规则三分类
- `suggest_bids(rows, rules)` —— 出调价建议
- `export_negatives(rows)` —— 导出否定词
- 复用 query-system 的 Excel 导出能力做报告输出

**人设 (Prompt) 要点**：你是亚马逊广告优化师，用数据 + 老板的人工规则给出可执行的调词/调价动作。
**判断规则 (Skill)**：把"日常调词控 ACOS 的人工规则"转成明确阈值（如 ACOS>X 且转化<Y → 降价/否定）；盈亏案例转成正反判断标准。

**需要的数据 (Data)**：
- 近 90 天完整 SP/SD/SB 广告报表
- 人工调词控 ACOS 规则
- 盈利/亏损广告正反案例
- 竞品流量关键词数据

**依赖**：数据加载层 (0.3)。
**输出物**：指标汇总表 + 好词/长尾/垃圾词分类 + 调价建议 + 否定词清单 + 拓词。
**验收标准**：导入真实报表 → 输出的三分类和调价建议与老板人工判断高度吻合；否定词清单可直接上传。**量化闸门**：先与老板对齐分类阈值（见 G3），并用 golden set（G1）验证三分类与人工标注的一致率达标（如 ≥ 90%）。

## #5 爆款 & 精品拆解智能体 【第三梯队 / 乙 / 难度：中】

**定位**：输入竞品 ASIN，自动扒出对方整套打法。靠已做好的 query-system。
**复用提示**：变体拆解、评价痛点多为**组装现有能力**——`models.py` 已有变体字段、`search_products` 已返回评论+情感分析，不必从零造，主要缺的是 0.5 的 `/api/product/{asin}` 路由。

**核心功能拆解**：
- 5.1 **整套 Listing 拆解** —— 标题/五点/A+/图/视频/关键词
- 5.2 **变体结构拆解** —— 父子 ASIN、变体维度（颜色/尺寸/套装）
- 5.3 **定价 & 配套配件识别** —— 价格带、赠品/配件搭配
- 5.4 **评价 & QA 痛点汇总** —— 抓评价+QA，聚类出用户痛点（调 query-system 的评论情感分析）
- 5.5 **运营模式分类** —— 判断对方是 精品 / 铺货 / 标品

**要开发的工具 (Tool)**：
- query-system 加 `/api/product/{asin}`（任务 0.5），`tools/amazon.py` 加对应 `get_product_by_asin`
- 复用 query-system 已有的评论情感/关键词分析
- `classify_operation_mode(product)` —— 精品/铺货/标品 判断

**人设 (Prompt) 要点**：你是竞品拆解专家，输出结构化的"对手打法报告"。
**判断规则 (Skill)**：精品/铺货/标品的判定标准、痛点聚类方法、盈利判定标准。

**需要的数据 (Data)**：
- 类目爆款 ASIN 清单
- 竞品完整 Listing + 评价（部分可由 query-system 实时抓）
- 精品盈利判定标准
- 同类滞销产品复盘

**依赖**：query-system（+0.5 新接口）、#1（拆解文案时可选查合规）。
**输出物**：竞品拆解报告（Listing/变体/定价/配件/痛点/运营模式）+ 结构化 JSON（供 #7 消费，见 G7）。
**验收标准**：输入竞品 ASIN → 输出完整拆解报告 + 痛点清单 + 运营模式判断。

## #6 市场数据分析智能体 【第三梯队 / 乙 / 难度：中】

**定位**：对类目/关键词做市场研判，判断蓝海红海。

**核心功能拆解**：
- 6.1 **搜索体量测算** —— 类目/关键词的搜索规模
- 6.2 **竞品竞争强度评估** —— 头部集中度、评论门槛、卖家数量
- 6.3 **价格利润区间划分** —— 高/中/低价格带及各带利润空间
- 6.4 **淡旺季 / 节日需求识别** —— 季节波动、节日峰值
- 6.5 **专利扎堆 / 高风险类目标记** —— 调用 #1 标风险

**要开发的工具 (Tool)**：
- `parse_keyword_market_data(excel)` —— 读关键词工具导出的市场数据
- 复用 query-system 抓各价位竞品样本
- 调用 #1 `flag_category_risk`

**人设 (Prompt) 要点**：你是市场分析师，输出"蓝海/红海"结论 + 依据。
**判断规则 (Skill)**：把"老板判断蓝海/红海的运营标准"转成可量化指标；淡旺季识别方法。

**需要的数据 (Data)**：
- 关键词工具导出的市场数据
- 各价位竞品样本
- 历年淡旺季销售记录
- 蓝海/红海判断标准

**依赖**：query-system、#1、数据加载层。
**输出物**：市场研判报告（体量/竞争强度/价格带/季节性/风险）+ 结构化 JSON（供 #7 消费，见 G7）。
**验收标准**：输入关键词/类目 → 输出研判报告，蓝海红海结论有量化依据。

## #7 AI 智能选品智能体 【第三梯队 / 甲 / 难度：高】

**定位**：整个选品链的"大脑"，综合前面几个智能体的结论给出选品决策。最复杂。

**核心功能拆解**：
- 7.1 **多维评分模型** —— 市场容量 / 竞争 / 毛利 / 侵权风险 / 改良空间，各维打分 + 加权总分
- 7.2 **差评挖掘改良方向** —— 从竞品差评提炼"可改良点"（调 query-system 评论分析）
- 7.3 **侵权风险排查** —— 调用 #1 合规
- 7.4 **毛利测算** —— 接入成本测算表算真实毛利
- 7.5 **标准化风险收益报告** —— 每个候选品出一张评分卡 + 结论

**要开发的工具 (Tool)**：
- `score_product(候选品, 权重, 门槛)` —— 评分模型（核心，甲写）
- `calc_gross_margin(成本表, 售价)` —— 毛利测算
- 编排：调用 #6 市场、#5 拆解、#1 合规 的结果作为评分输入（靠 0.2 注册表）

**人设 (Prompt) 要点**：你是选品决策官，每一维打分都要给可解释的依据，不拍脑袋。
**判断规则 (Skill)**：各维评分标准与权重、选品硬性门槛（最低毛利、禁做品类）、侵权造型/功能排查规则、成功&失败复盘转成的经验规则。

**需要的数据 (Data)**：
- 完整成本测算表
- 选品硬性门槛（最低毛利、禁做品类）
- 侵权造型/功能排查规则
- 过往成功 & 失败选品复盘
- 主营赛道客群偏好

**依赖**：#1、#5、#6、query-system、成本表、注册表 (0.2)。**依赖最重，放第三梯队末。**
**输出物**：选品评分卡 + 改良建议 + 风险收益报告。
**验收标准**：输入候选品 → 输出各维评分（可解释）+ 总分 + 改良点 + 风险收益结论；不达门槛能自动淘汰。**前置**：各维权重与硬性门槛须先与老板对齐为明确数字（G3）；编排 #1/#5/#6 时遵守失败降级策略（G4）。

## #8 评价 & 站内客服智能体 【第四梯队 / 甲 / 难度：高（需 API）/ 后期】

**定位**：自动处理买家消息、差评、催评。需要亚马逊 SP-API 授权，放后期。
**安全**：买家消息含 PII、SP-API token 为密钥，遵守 G5（密钥进 env/密钥库、PII 不落公共缓存、访问受控）。授权申请周期长，**Day1 即提交**。

**核心功能拆解**：
- 8.1 **差评原因归类** —— 把差评自动分类（质量/物流/描述不符/客服…）
- 8.2 **合规安抚/补偿话术匹配** —— 按差评类型匹配退换/补偿话术（合规，调 #1）
- 8.3 **催评邮件生成** —— 批量生成合规催评邮件
- 8.4 **绩效申诉模板生成** —— 违规/绩效问题的申诉模板
- 8.5 **高频问题 → FAQ 沉淀** —— 从历史买家咨询聚类出 FAQ

**要开发的工具 (Tool)**：SP-API 对接（读买家消息/评价）、话术匹配、邮件模板生成。
**人设 (Prompt) 要点**：你是合规客服，回复必须符合亚马逊站内消息规则，不诱导好评、不留联系方式。
**判断规则 (Skill)**：差评归类标准、话术库映射、站内消息合规红线。

**需要的数据 (Data)**：

- 全套客服话术模板
- 买家高频咨询问题
- 亚马逊站内消息合规限制

**依赖**：SP-API 授权、#1 合规、话术库。
**输出物**：差评回复草稿、催评邮件、申诉模板、FAQ。

**要开发的工具 (Tool)**：SP-API 对接（库存/订单/绩效）、毛利计算器、调价规则引擎、告警阈值判断。
