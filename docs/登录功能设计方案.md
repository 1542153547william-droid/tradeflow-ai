# 登录功能设计方案

## Context

TradeFlow-AI 的 Web 层（`web/app.py` + `web/static/prototype.html`）目前是"伪多租户"：数据库 schema（`users`/`stores`/`opportunities`/`chat_sessions` 等）都按 `user_id`+`store_id` 做了隔离设计，但从未真正启用——前端从不发用户身份，所有接口的 `x_tradeflow_user` 参数永远吃到默认值 `"default"`，唯一的"认证"是一个可选的全局 HTTP Basic Auth（一套共享用户名密码，不区分用户，见 `web/app.py:58-75`）。

现在要把它变成真正按账号隔离的登录系统。已与用户确认的产品决策：
- **不做自助注册**，账号由管理员用脚本手动开通（当前是内部工具，非公开产品）。
- **Cookie Session** 维持登录态（同源静态页 + FastAPI，httpOnly cookie 最简单也最安全，不用前端管 token）。
- **首个新账号自动继承现有 `default` 用户的历史数据**（机会点、导入记录、聊天记录等，保证已经在跑的功能不丢失）。

## 设计

### 1. 数据库改动（`web/database.py`）
- `users` 表新增 `username TEXT UNIQUE`、`password_hash TEXT` 两列。SQLite 不支持 `ADD COLUMN IF NOT EXISTS`，用 `PRAGMA table_info` 查列是否存在、不存在则 `ALTER TABLE ... ADD COLUMN` 的小 helper，放在 `init_db()` 里，和现有"建表用 `IF NOT EXISTS`"的幂等风格保持一致。
- 新增 `sessions` 表：`token_hash TEXT PRIMARY KEY, user_id TEXT NOT NULL, created_at, expires_at`，外键指向 `users(id)`。存的是 token 的 sha256 哈希，不存明文，DB 泄露不等于会话泄露。

### 2. 新模块 `web/auth.py`
- `hash_password` / `verify_password`：用 `bcrypt`（新依赖，`requirements.txt` 加一行）。
- `create_session(user_id) -> str`：生成 `secrets.token_urlsafe(32)`，哈希后连同过期时间写入 `sessions` 表，返回明文 token 给调用方去种 cookie。
- `resolve_session(token) -> str | None`：按哈希查 `sessions`，校验未过期，返回 `user_id`。
- `delete_session(token)`。
- FastAPI 依赖 `current_user(request: Request) -> str`：从 cookie 取 token → `resolve_session`，拿不到或过期就 `raise HTTPException(401)`。

### 3. 配置（`config/settings.py`）
该文件是纯 `@dataclass` + `os.environ.get(...)` 风格（不是 pydantic），新增字段按同样写法加：`session_cookie_name`（默认 `tf_session`）、`session_ttl_days`（默认 30，环境变量 `TRADEFLOW_SESSION_TTL_DAYS` 可调）、`secure_cookies`（默认 `True`，本地 http 调试可用 `TRADEFLOW_SECURE_COOKIES=0` 关掉，跟现有 `TRADEFLOW_BASIC_USER` 这类命名风格一致）。`basic_auth_user`/`basic_auth_password`/中间件里对应的读取一并删除（§5）。

### 4. 认证接口（`web/app.py`）
- `POST /api/auth/login` `{username, password}` → 校验 → `create_session` → `response.set_cookie(httponly=True, samesite="lax", secure=settings.secure_cookies, max_age=...)` → 返回 `{id, name}`。
- `POST /api/auth/logout` → 删 session 行 + `delete_cookie`。
- `GET /api/auth/me` → `Depends(current_user)`，成功返回 `{id, name}`，失败让依赖自然抛 401（前端用它判断是否已登录）。

### 5. 全局网关（`require_api_token` 中间件，`web/app.py:58-75`）
- **已确认：直接移除**现有的全局共享 Basic Auth（`TRADEFLOW_BASIC_USER`/`TRADEFLOW_BASIC_PASSWORD` 这两个环境变量随之失效）。登录接口本身已提供访问入口保护，两套认证并存是冗余模型。
- 新增门禁：除 `/api/auth/login`、静态资源（`/`、`/static/*`、经典页）外，所有 `/api/*` 请求必须带有效 session cookie，否则 401。直接在中间件里调 `auth.resolve_session`，做统一拦截，不依赖每个 handler 都记得加依赖（防漏）。

### 6. 替换现有的"信任 Header"取用户逻辑
- `app.py` 里大量 `x_tradeflow_user: str = Header(default="default")` 参数（`stores`/`create_store`/`chat_sessions` 系列/`opportunities` CRUD/`imports`/`optimization_ads`/`customers`/`opportunities_suggest`/`chat`/`chat_stream`，约 15 处）统一换成 `Depends(current_user)` 取真实登录用户 id。
- `x_tradeflow_store` header 保留，继续用来切店铺；店铺归属校验复用现有 `_scope`（`web/app.py:78-88`）里 `SELECT 1 FROM stores WHERE id=? AND user_id=?` 的模式，只是 `user_id` 现在来自 session 而不是客户端自报的 header。
- 这一批是纯机械的多处替换，实现阶段适合交给自动化工具批量做，替换规则和"哪些端点保持公开"的清单明确后再逐条核对合并。

### 7. 管理员开号脚本（新增 `web/manage.py`）
- `python -m web.manage create-user <username> <password> [--name 显示名]`。
- 逻辑：`init_db()` → 若这是第一个非 `default` 用户，创建成功后把 `stores`/`opportunities`/`import_batches`/`imported_rows`/`chat_sessions`/`audit_log` 里 `user_id='default'` 的行整体 `UPDATE` 成新用户 id（一次性迁移，只在建第一个账号时触发）；之后再开的账号不再触发迁移，直接是空账号。

### 8. 前端（`web/static/prototype.html`）
- 现有 boot 逻辑在 `fetch('/api/stores')...`（约第 1383 行）之前，先 `GET /api/auth/me`：401 就隐藏 `.app`、展示一个新的全屏登录卡片（用户名/密码输入框 + 登录按钮 + 错误提示，样式沿用现有 CSS 变量），提交 `POST /api/auth/login`，成功后重新走 boot 流程；成功则直接进入现有界面。
- `.topbar`（约第 380-390 行）加一个"退出登录"图标按钮，点击 `POST /api/auth/logout` 后重载页面。
- fetch 都是同源请求，浏览器会自动带上 cookie，不需要改任何现有 fetch 调用的 header。

## 涉及文件
- `web/database.py`（schema）
- `web/auth.py`（新建）
- `web/manage.py`（新建）
- `web/app.py`（登录三个接口、中间件门禁、~15 处 Header→Depends 替换）
- `config/settings.py`（新增配置项）
- `requirements.txt`（加 `bcrypt`）
- `web/static/prototype.html`（登录覆盖层 + 退出按钮 + boot 逻辑接入 `/api/auth/me`）

## 验证
1. `python -m web.manage create-user admin <密码>`，确认历史 `default` 数据（机会点/导入记录/对话）迁移到 admin 名下。
2. 起服务访问 `/`：未登录应看到登录卡片而非应用界面；密码错报错；密码对进入应用且历史数据都在。
3. `curl` 不带 cookie 直接打 `/api/opportunities` 应返回 401，证明门禁生效。
4. 退出登录后页面应回到登录卡片。
5. 再建第二个账号，确认它是空的、看不到第一个账号的店铺/数据（隔离性验证）。
6. 跑 `tests/test_p0.py` 等现有测试，若因为端点现在要求登录而挂掉，需要给测试加一个"先登录拿 cookie"的 fixture（作为实现阶段的已知后续工作标注）。
7. 提交前对 diff 做一次炸点审查（会话并发、cookie 安全属性、SQL 注入等）。
