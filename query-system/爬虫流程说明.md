# Amazon 查询系统 · 爬虫流程说明

> 说明当前项目爬虫通道的工作流程、反爬手段，以及实现与 PRD 需求的差距。
> 涉及文件：`backend/app/datasources/scraper_source.py`、`backend/app/services/search_service.py`、`backend/app/config.py`。

---

## 一、总体定位：爬虫是「回退通道」，不是主力

系统采用三级数据源，爬虫夹在中间（`search_service.py:26-34`）：

```
API（合规第三方，如 Rainforest）  →失败→  Scraper（Playwright 爬虫）  →失败→  Mock（假数据兜底）
```

- **选谁由 `resolved_source` 决定**（`config.py:48-57`）：有 `api_key` 走 API；否则若 `scraper_enabled` 走爬虫；再否则 mock。
- **缓存优先**：请求先查 SQLite 缓存（key = 关键词 + 站点 + topN，TTL 24h），命中直接返回，不触发任何抓取（`search_service.py:52-55`）。
- **失败即回退**：任一数据源抛 `DataSourceError` 或任何异常 → 静默回退到下一个，全部失败才报错（`search_service.py:79-89`）。

### 数据流总览

```
请求
 └─> 查缓存 ── 命中 ──> 直接返回
        │
      未命中
        └─> 数据源链：api → scraper → mock
                          │
                       (scraper)
                          └─> 启动浏览器 → 开 context → 抓 SERP → 解析卡片
                              → 补全品牌(详情页) → 抓评论 → 写缓存 → 返回
```

## 二、爬虫内部流程（ScraperSource）

### 1) 浏览器启动（懒加载、单例）
首次抓取时才启动 Chromium，用**双检锁** `_launch_lock` 保证并发协程只起一个实例、避免泄漏（`scraper_source.py:104-127`）。
启动参数带 `--disable-blink-features=AutomationControlled` 抹掉自动化特征。

### 2) 每次请求开新 context（反指纹）
`_new_page`（`scraper_source.py:129-138`）每次：
- 随机选 User-Agent
- 固定 `en-US` locale、1366×900 视口、`Accept-Language` 头
- 注入 `_STEALTH_JS`：抹掉 `navigator.webdriver`，伪造 `chrome` / `languages` / `plugins`

### 3) 搜索抓取 `_search_once`（`scraper_source.py:158`）

1. 拼 URL `https://www.{站点}/s?k=关键词` → `goto`（`wait_until="domcontentloaded"`）
2. **随机延时** `_throttle`（1.5–4s，`scraper_source.py:140`）
3. **拦截检测** `_is_blocked`：标题含 `robot`/`captcha`/`sorry` 或出现验证码表单 → 抛错触发回退（`scraper_source.py:362`）
4. 等结果容器渲染，遍历 `div[data-component-type='s-search-result']` 卡片，逐条：
   - 跳过空 / 重复 ASIN
   - 跳过 **Sponsored 广告位**（`_is_sponsored`）→ 保证排名反映自然结果
   - 解析字段：标题、价格 + 币种、划线价与折扣、评分、评论数、Prime、图片、position
5. 解析到 0 个产品 → 抛错（判定为页面结构变化 / 被拦截）

#### 价格与币种解析
- `_parse_money`（`scraper_source.py:46`）兼容美式 `1,234.56` 与德式 `1.234,56` 两种千分位/小数点格式。
- `_parse_currency`（`scraper_source.py:79`）优先读文本内货币代码/符号，`$` 有歧义（USD/CAD/AUD）时按站点 `_MARKET_CURRENCY` 兜底。

### 4) 品牌补全（可选，进详情页）
搜索页品牌拿不稳，故 `scraper_fetch_brand` 开启时，对每个 ASIN 打开 `/dp/{asin}` 详情页，从 byline 或商品概览表取品牌；**并发限 3**（Semaphore），失败不影响主结果（`scraper_source.py:239-278`）。

### 5) 评论抓取
由 `SearchService` 触发（`search_service.py:91-109`）：
- 只取**前 5 个产品**，每个最多 40 条，并发限 3
- `fetch_reviews` 打开 `/product-reviews/{asin}` 解析（`scraper_source.py:280`）

### 6) 容错与清理
- 搜索失败**重试 1 次**（换新 context）再放弃（`scraper_source.py:145-156`）
- 每次请求 `finally` 关 context；`close()` 关浏览器与 Playwright

## 三、反爬手段小结

| 手段 | 位置 |
|---|---|
| 随机 User-Agent 轮换 | `_new_page` |
| 随机延时 1.5–4s | `_throttle` |
| Stealth JS 指纹伪装 | `_STEALTH_JS` |
| 关闭自动化特征 | 启动参数 `--disable-blink-features=AutomationControlled` |
| 详情页/评论并发上限（Semaphore=3） | `_fill_brands` / `_gather_reviews` |
| 拦截即放弃回退（**不做自动破解**） | `_is_blocked` |

> 说明：当前策略是「检测到验证码就放弃并回退」，**未做自动破解**——与 PRD V1.1「规避触发而非暴力突破」的合规方向一致。

## 四、实现 vs PRD 需求的差距

| 需求点（PRD） | 当前实现 | 状态 |
|---|---|---|
| 精准打标 Organic / SP / SB / SBV 并保留广告坑位 | 只区分「是不是广告」，广告位被**直接跳过**，未细分 | ❌ 差距明显 |
| 住宅代理 IP 池 / 封禁熔断降级 | 无代理，仅单机延时 + UA 轮换 | ❌ 未实现 |
| BSR（大/小类目排名） | 未抓取 | ❌ 未实现 |
| 预估月销量（BSR 模型折算） | 未实现 | ❌ 未实现 |
| 父子体（Parent/Child ASIN） | 未抓取 | ❌ 未实现 |
| Buy Box 归属 / 促销标签 | 未抓取 | ❌ 未实现 |
| 模拟邮编（本地运费/Prime） | 未实现（仅取页面默认 Prime 标识） | ❌ 未实现 |
| 站点选择、价格/币种、评分、评论数、Prime | 已实现 | ✅ 已实现 |
| 批量关键词 | 单次单词，服务层未做批量编排 | ⚠️ 部分 |

---

## 五、后续改造建议（优先级）

1. **广告位细分打标（P0）**：保留广告卡并区分 SP/SB/SBV，避免"自然排名"口径失真。
2. **代理池接入 + 封禁熔断（P0）**：规模化抓取的前提，否则极易被封。
3. **补齐关键字段（P1）**：BSR、Buy Box、促销、父子体。
4. **模拟邮编（P1）**：还原本地运费/Prime，保证数据可复现。
5. **预估销量模型（P2）**：依赖 BSR→销量映射模型，需先验证准确度。
